## 설정하기

`student/` 디렉토리 내의 파일만 수정합니다. 그 외 파일은 변경하지 않습니다.

### conf/flink-conf.yaml

파일 문법: `키: 값` 형식. `#`으로 주석 처리된 줄에서 `#`을 제거하고 값을 입력합니다.

```yaml
# execution.checkpointing.interval:
execution.checkpointing.interval: 10000
```

**Checkpointing**

Flink는 스트리밍 처리 중 상태(집계값, 세션 정보 등)를 메모리에 유지합니다. 분산 시스템에서 노드 장애는 언제든 발생할 수 있으며, 장애 발생 시 메모리 상태는 유실됩니다. Flink는 이를 방지하기 위해 주기적으로 상태 전체를 외부 경로에 스냅샷으로 저장하고, 장애 발생 시 가장 최근 스냅샷에서 복구합니다.

동일한 이벤트가 두 번 집계되어서는 안 되는 경우, mode는 어떻게 설정해야 할까요?

**이 설정이 없으면** JobManager가 잡 제출을 거부하며 시나리오 01이 즉시 실패합니다.

| 키 | 설명 | 형식 예시 |
|----|------|-----------|
| `execution.checkpointing.interval` | 스냅샷 주기 (ms) | `10000` (= 10초) |
| `execution.checkpointing.mode` | 중복 처리 허용 여부 | `EXACTLY_ONCE` / `AT_LEAST_ONCE` |
| `state.checkpoints.dir` | 스냅샷 저장 경로 | `file:///data/flink-checkpoints` |
| `state.backend` | 상태 저장 방식 | `hashmap` (메모리) / `rocksdb` (디스크) |

**병렬성**

Flink는 잡을 여러 태스크로 분할하여 TaskManager의 슬롯에 분배합니다. 가용 슬롯 총합은 `TM 수 × TM당 슬롯 수`이며, parallelism이 이 값을 초과하면 잡이 PENDING 상태에서 멈춥니다. 이 클러스터의 TaskManager는 3대이며, TM당 메모리는 1728m입니다.

**잘못 설정하면** parallelism이 가용 슬롯보다 크게 설정될 경우 잡이 `SCHEDULED` 상태에서 진행되지 않고 timeout으로 실패합니다.

| 키 | 설명 | 힌트 |
|----|------|------|
| `taskmanager.numberOfTaskSlots` | TM 1대당 슬롯 수 | 슬롯이 많을수록 메모리를 잘게 나눕니다. `1` 또는 `2` 권장 |
| `parallelism.default` | 기본 병렬 태스크 수 | `TM 수(3) × numberOfTaskSlots` 이하로 설정 |

---

### conf/spark-defaults.conf

파일 문법: `키    값` 형식으로 공백으로 구분합니다.

```
# spark.default.parallelism
spark.default.parallelism    6
```

**Shuffle Partition**

groupBy 또는 join을 실행하면 Spark는 동일한 키의 데이터를 하나의 파티션으로 집결시킵니다. 이 과정이 셔플이며, 파티션 하나당 태스크 하나가 Executor에 할당됩니다. 파티션 수가 클러스터 코어 수보다 지나치게 많으면 실제 연산보다 태스크 스케줄링 오버헤드가 더 커집니다. 이 클러스터는 worker 3대 × 코어 2개 = 총 6코어입니다.

**200으로 두면** 코어 6개가 태스크 200개를 처리하면서 스케줄링 오버헤드가 급증하여 시나리오 04가 실패합니다.

| 키 | 현재 값 | 형식 예시 |
|----|---------|-----------|
| `spark.sql.shuffle.partitions` | `200` → 변경 필요 | `spark.sql.shuffle.partitions    6` |
| `spark.default.parallelism` | 미설정 | `spark.default.parallelism    6` |

### 03 — stream-vs-batch

같은 데이터를 Flink와 Spark 양쪽으로 처리하여 **latency와 throughput을 비교**합니다.

실행 후 `results/result.json`에서 두 엔진의 수치를 확인합니다.

- Flink는 latency가 짧습니다. Spark는 데이터를 모아서 한 번에 처리하고, Flink는 입력 즉시 처리합니다.
- throughput도 함께 검토합니다. 대용량 야간 배치 처리라면 어느 엔진이 더 유리할지 생각해 봅니다.

**참고**: `spark_job_time_ms`에는 Spark 잡을 시작할 때마다 발생하는 JVM 부트업 오버헤드가 포함됩니다 (통상 수 초). Flink는 TaskManager가 이미 실행 중인 상태에서 잡만 제출하므로 이 오버헤드가 없습니다. 이는 Spark가 본질적으로 느리다는 의미가 아니라, 배치 엔진은 매번 잡을 새로 시작하는 구조이기 때문에 짧은 작업에서 불리하다는 의미입니다.

**REPORT.md `03` 섹션**에서 두 관점을 모두 정리합니다.

### 04 — spark-shuffle

shuffle partition 수가 성능에 미치는 영향과 Spark의 **DAG 구조**를 함께 분석합니다.

이 시나리오는 **baseline(200) + 본인 설정값**으로 두 번 spark-submit을 실행합니다. result.json의 `baseline_time_ms`(200 기준)와 `job_time_ms`(설정값 기준)를 비교할 수 있습니다.

**진행 순서:**

1. `spark.sql.shuffle.partitions`를 클러스터 코어 수에 맞게 설정한 후 `./scripts/cluster.sh init` → `./run_all.sh --scenarios 4` 실행.
2. **http://localhost:8080** → 실행 중인 잡 클릭 → **Stages** 탭에서 태스크 수가 `shuffle.partitions` 값과 일치하는지 확인합니다.
3. **DAG Visualization**을 열어 stage가 어떻게 분할되는지 확인합니다. groupBy 전후로 stage가 분리되는 지점이 셔플 경계입니다.

**참고**: `speedup_factor`가 1보다 작게(예: 0.6) 측정될 수 있습니다. 이 데이터셋 규모에서는 두 번째 spark-submit이 JVM 재기동 비용을 다시 부담하고 캐시도 초기화되어, 단순 실행 시간 비교만으로는 partition 효과가 측정되지 않을 수 있습니다. 핵심 학습 포인트는 "파티션 수가 태스크 수를 결정한다"는 사실이며, Stages 탭의 태스크 수 차이로 이를 직접 확인할 수 있습니다.

**시나리오 통과 조건**: `shuffle.partitions`가 8 이하여야 합니다 (200으로 두면 자동 실패).

**REPORT.md `04` 섹션**에 partition 수 비교와 DAG 관찰 결과를 작성합니다.

### 05 — pipeline

Kafka → Flink → Spark 전체 파이프라인을 end-to-end로 실행합니다.

두 엔진의 **fault tolerance 접근 방식 차이**도 함께 정리합니다.

- Spark는 어떤 방식으로 fault tolerance를 제공하나요? (힌트: RDD lineage)

**REPORT.md `05` 섹션**에서 두 엔진의 fault tolerance 방식을 비교하여 정리합니다.
